Data Normalisation & Enrichment Pipeline
Merges all source data into a unified schema. Deduplicates. Adds enrichment fields. Writes to Airtable via API.
1 · Normalise
Map each source's fields to a shared schema (name, category, lat/lng, description, photo_url, source_ref). Strip chains & multiples.
2 · Deduplicate
Fuzzy match on name + postcode. Flag likely duplicates for human review rather than auto-deleting. Score confidence 0–1.
3 · Enrich
Auto-populate missing fields by cross-referencing sources. Generate a short AI-draft bio (Claude API) as a starting point for editors.
normalise.py dedup.py enrich.py airtable_push.py

Trigger Logic
Scrape scripts (scrape_*.py) run manually or on a weekly cron (e.g. GitHub Actions — free). Normalise + deduplicate run immediately after each scrape. Airtable→Supabase sync (airtable_to_supabase.py) runs on a webhook trigger when an editor changes a record status to PUBLISHED.
Curation Is The Product
Nothing in the scrape → process lanes reaches the live site without an editor decision. Guides can flag and annotate but not approve. Only named editors can set status to PUBLISHED. This is enforced at the Airtable permission level and validated by validate.py before the Supabase push.
Scaling Path
At low volume: run scripts locally, use Airtable free tier (≤1k records), Supabase free tier. When you outgrow Airtable, the CMS can be replaced with a custom Supabase admin panel. The frontend (index.html) never needs to change — only the data source.
